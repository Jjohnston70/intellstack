# Bearing Check Quality Review Checklist

**Version:** 1.0
**Last Updated:** January 29, 2026
**Purpose:** Ensure consistent quality standards for Bearing Check decision analyses

---

## Overview

This checklist ensures every Bearing Check analysis meets quality standards. Use it:
1. **During Analysis:** Self-check after completing each checkpoint
2. **Before Finalizing:** Review entire analysis before making decision
3. **After Decision:** Retrospective review to improve future analyses

---

## Pre-Analysis Checklist

Before starting a Bearing Check, verify:

### Decision Clarity
- [ ] Decision is stated as a clear, specific question
- [ ] Stakes are identified (time, money, opportunity cost)
- [ ] Time horizon for decision is defined
- [ ] Reversibility is understood

### Scope Selection
- [ ] Appropriate check type selected (Full / Quick / Single)
- [ ] Stakes justify the analysis depth chosen
- [ ] Time allocated matches check type

### Information Readiness
- [ ] Basic information about the decision is available
- [ ] Access to research tools if needed
- [ ] No obvious blockers to completing analysis

**Pre-Analysis Quality Gate:**
If any of the above are unclear, clarify before proceeding.

## Checkpoint-by-Checkpoint Quality Checks

### Checkpoint 1: Ground Truth

**Completion Criteria:**
- [ ] Base rate estimated (not just "high" or "low"—needs a number or range)
- [ ] Source of base rate documented
- [ ] Base rate is for similar situations (not general population)
- [ ] Multiple sources considered if available

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Is the base rate specific to this decision type? | [ ] |
| Could someone else verify this base rate? | [ ] |
| Am I using outliers or typical outcomes? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Vague estimate ("pretty good odds") | Quantify: "40-60% success rate" |
| Using outlier examples | Find population-level data |
| "My situation is different" | Test: How is it actually different? |

**Red Flags That Fail This Checkpoint:**
- [ ] Cannot estimate base rate at all
- [ ] Only knows success stories
- [ ] Base rate comes from motivated source (salesperson, promoter)

---

### Checkpoint 2: Graveyard Recon

**Completion Criteria:**
- [ ] At least 3 failure examples identified
- [ ] Failure patterns documented (not just "they failed")
- [ ] Causes of failure distinguished from symptoms
- [ ] Relevance to my situation assessed

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Did I actively search for failures? | [ ] |
| Do I understand WHY they failed? | [ ] |
| Could their failure patterns apply to me? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| "They did it wrong" | Ask: What specifically did they do wrong? |
| Can't find failures | Search: "[topic] failure" or "[topic] post-mortem" |
| Dismissing failures as irrelevant | Test: What makes my situation immune? |

**Red Flags That Fail This Checkpoint:**
- [ ] Cannot name any failures
- [ ] Believes all failures are due to others' incompetence
- [ ] Only researched success stories

---

### Checkpoint 3: Mechanism Check

**Completion Criteria:**
- [ ] Underlying mechanism identified (not just behavior)
- [ ] At least 2 alternative explanations considered
- [ ] Alternatives tested against evidence
- [ ] Correlation vs. causation addressed

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Can I explain WHY this works, not just THAT it works? | [ ] |
| Have I considered alternative explanations? | [ ] |
| Is my explanation falsifiable? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Tautological explanation ("they succeeded because they're good") | Ask: What specifically did they do that others didn't? |
| Copying visible behavior | Ask: Is this behavior the cause or effect of success? |
| Single-factor attribution | Generate multiple hypotheses |

**Red Flags That Fail This Checkpoint:**
- [ ] Cannot separate behavior from mechanism
- [ ] Explanation is circular
- [ ] Attributes success to easily-copied surface behavior

---

### Checkpoint 4: Fixed Points

**Completion Criteria:**
- [ ] Success factors categorized as durable vs. situational
- [ ] Timing/trend dependency assessed
- [ ] Control vs. outside-control factors identified

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Which factors would still matter in 5 years? | [ ] |
| Am I depending on things I can control? | [ ] |
| If conditions change, does the strategy still work? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| All factors seem durable | Stress test: What would have to change to break this? |
| Relying on timing | Ask: Would this work 6 months earlier or later? |
| Core factors outside my control | Identify what I CAN control |

**Red Flags That Fail This Checkpoint:**
- [ ] Strategy depends entirely on current conditions
- [ ] No durable competitive advantage identified
- [ ] Core factors are outside my control

---

### Checkpoint 5: Odds Assessment

**Completion Criteria:**
- [ ] Probability expressed as range, not point estimate
- [ ] Success scenario defined
- [ ] Failure scenarios defined (multiple)
- [ ] Downside explicitly considered

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Am I using ranges, not certainties? | [ ] |
| Do I know what failure looks like? | [ ] |
| Have I considered "what if I'm wrong"? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Point estimates ("80% chance") | Use ranges: "60-80% chance" |
| Speaking in certainties | Add uncertainty: "under these conditions" |
| Only considering success | Force failure scenario: "This fails if..." |

**Red Flags That Fail This Checkpoint:**
- [ ] Cannot articulate failure conditions
- [ ] Believes probability doesn't apply
- [ ] Rejects premise that this could fail

---

### Checkpoint 6: Constraint Fit

**Completion Criteria:**
- [ ] All major constraints assessed (time, money, emotional, etc.)
- [ ] Strategy requirements compared to my reality
- [ ] Gaps identified and addressed or accepted
- [ ] Emotional cost considered, not just financial

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Did I complete the full constraint table? | [ ] |
| Am I honest about my constraints? | [ ] |
| Have I considered emotional bandwidth? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Assuming I can change constraints to fit | Accept constraints, adapt strategy |
| Copying someone else's strategy without their resources | Modify for my resources |
| Ignoring emotional cost | Ask: Can I sustain this through difficulty? |

**Red Flags That Fail This Checkpoint:**
- [ ] Major constraint mismatch ignored
- [ ] Borrowed strategy from someone with different resources
- [ ] Emotional cost not considered

---

### Checkpoint 7: Staged Advance

**Completion Criteria:**
- [ ] Stage 1 (minimum viable test) defined
- [ ] Success signals for each stage defined
- [ ] Abort signals defined
- [ ] Decision points identified

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Can I test this with minimal commitment? | [ ] |
| Do I know what evidence would change my mind? | [ ] |
| Is there reversibility built in? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Wants all-in immediately | Define smallest possible test |
| No reversibility | Build decision points into plan |
| "Staged approach shows lack of commitment" | Staged approach IS the commitment to learning |

**Red Flags That Fail This Checkpoint:**
- [ ] No staged plan possible
- [ ] Cannot articulate what would change mind
- [ ] All-or-nothing thinking

---

### Checkpoint 8: Pre-Mortem

**Completion Criteria:**
- [ ] Failure narrative written in past tense
- [ ] Specific failure causes identified
- [ ] Survivability assessed (financial, emotional, professional)
- [ ] Repeatability evaluated

**Quality Questions:**
| Question | Pass? |
|----------|-------|
| Did I actually imagine failure? | [ ] |
| Is the failure scenario realistic? | [ ] |
| Can I survive this failure? | [ ] |

**Common Issues:**
| Issue | Fix |
|-------|-----|
| Hasn't imagined failure | Write the failure story: "It's one year later and this failed because..." |
| Failure narrative too vague | Add specific causes and consequences |
| Survivability not assessed | Rate: Easily survivable / Damaging / Catastrophic |

**Red Flags That Fail This Checkpoint:**
- [ ] Cannot imagine failure
- [ ] Failure would be catastrophic
- [ ] Strategy is one-time gamble, not repeatable

## Synthesis Quality Checklist

Before finalizing the analysis:

### Summary Quality
- [ ] Each checkpoint has a clear finding
- [ ] Flag levels (green/yellow/red) assigned
- [ ] Red flags are not rationalized away
- [ ] Yellow flags are acknowledged

### Recommendation Quality
- [ ] Recommendation is clear (GO / CONDITIONAL GO / NO-GO / DEFER)
- [ ] Confidence level stated
- [ ] Rationale is specific (not "feels right")
- [ ] Conditions stated if CONDITIONAL GO
- [ ] Information gaps stated if DEFER

### Next Action Quality
- [ ] Next action is specific and actionable
- [ ] If GO: Next skill identified (Direction Protocol, World Model Mapper, etc.)
- [ ] Timeline for next action clear

---

## Red Flag Count Guidelines

| Red Flags | Typical Recommendation |
|-----------|----------------------|
| 0-1 | GO (proceed with monitoring) |
| 2 | CONDITIONAL GO (address conditions) |
| 3+ | NO-GO or DEFER |

**Exception:** Even 1 red flag can justify NO-GO if:
- Red flag is in Ground Truth (base rate is terrible)
- Red flag is in Constraint Fit (fundamental mismatch)
- Red flag is in Pre-Mortem (failure is catastrophic)

---

## Post-Decision Review Checklist

After executing the decision (win or lose), review:

### Accuracy Assessment
- [ ] Did the decision outcome match the prediction?
- [ ] Were flags accurate predictors?
- [ ] Did ignored warnings come true?

### Learning Capture
- [ ] What would I do differently?
- [ ] Which checkpoint was most valuable?
- [ ] Which checkpoint did I rush?

### Documentation
- [ ] Outcome recorded in decision log
- [ ] Lessons learned documented
- [ ] Case study potential identified

## Common Quality Failures

### Failure 1: Confirmation Bias

**Symptom:** Analysis supports what you wanted to do anyway

**Signs:**
- All checkpoints somehow support GO
- Red flags rationalized
- Evidence selectively gathered

**Fix:**
- Assign devil's advocate role
- Force yourself to argue the opposite
- Have someone else review

### Failure 2: Rush Analysis

**Symptom:** Checkpoints completed too quickly

**Signs:**
- Estimates without research
- Single-source base rates
- Vague failure scenarios

**Fix:**
- Set minimum time per checkpoint
- Require sources for key estimates
- Use checklist before finalizing

### Failure 3: Stopping at Surface

**Symptom:** Checkpoints completed but not deeply

**Signs:**
- Mechanism is just behavior
- Failures dismissed as "they did it wrong"
- Constraints not honestly assessed

**Fix:**
- Ask "why" until mechanism is clear
- Force specificity in failure analysis
- Complete full constraint table

### Failure 4: Ignoring Red Flags

**Symptom:** Red flags identified but recommendation ignores them

**Signs:**
- "Despite the red flags, I recommend GO"
- Red flags characterized as "manageable"
- Excitement overrides analysis

**Fix:**
- Hard rule: 3+ red flags = NO-GO
- Sleep on decisions with red flags
- Get outside perspective

---

## Quick Quality Check (5 minutes)

For time-pressured reviews, verify:

1. [ ] Is there a base rate? (Not just "good" or "bad")
2. [ ] Did I search for failures? (Not just successes)
3. [ ] Do I understand the mechanism? (Not just behavior)
4. [ ] Did I quantify odds? (Not certainties)
5. [ ] Did I do the pre-mortem? (Actually imagine failure)

If NO to any of these, go back and complete properly.
