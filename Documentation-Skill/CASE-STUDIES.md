## Case Study 1: Command Center Build Service Document

### Context

**Situation:** TNDS needed a professional service document describing the Command Center Build offering for use in sales conversations and on the website.

**Request:** "Create a service document for Command Center Build"

**Source Skill:** None (new document creation)

**Audience:** Prospective clients (business owners)

---

### Process

#### Intake

**Clarification Questions Asked:**
- Who reads this? → Prospective clients, business owners
- Format needed? → Word (for sharing, printing)
- What action should reader take? → Understand offering, schedule a call

**Template Selected:** Client-Facing Service Document

**Information Gathered:**
- Pricing: $2,500-$5,000
- Timeline: 2-4 weeks
- Deliverables: Operational visibility system, Command modules
- ICP: 5-20 person operations, field service or admin-heavy

---

#### Development

**Structure Applied:**
1. Title and subtitle
2. What This Engagement Is
3. Who This Is For
4. Problems This Removes
5. What You Get
6. Scope & Pricing
7. Prerequisites
8. What Comes Next

**Key Writing Decisions:**

| Decision | Reasoning |
|----------|-----------|
| Lead with outcome | "See what's happening in your operation in real-time" |
| Use client voice for pain points | "I can't take a day off without things falling apart" |
| Specific deliverables | Named actual modules: data-command, analytics-command |
| Clear pricing | "$2,500-$5,000" with scope factors explained |

**Voice Check Points:**
- Changed "leverages data integration" → "connects your systems"
- Changed "enhances visibility" → "shows you what's happening"
- Added specific numbers: "real-time" instead of "improved"

---

#### Quality Check

**Content Quality:**
- [x] Direct, no-fluff voice ✓
- [x] Outcomes-focused ✓
- [x] Specific and concrete ✓
- [x] No jargon ✓

**Brand Quality:**
- [x] Logo in header ✓
- [x] Correct colors ✓
- [x] Footer formatted ✓

---

#### Outcome

**Final Document:** `TNDS_Service_CommandCenterBuild_2026-01-15.docx`

**Time Spent:** 35 minutes

**Result:** Document approved, now used in sales process

---

### Lessons Learned

1. **Client voice matters** — Writing pain points in first person ("I can't...") resonated more than third person
2. **Specific beats general** — Naming actual modules more credible than "various components"
3. **Pricing transparency** — Range with factors explained reduces follow-up questions

## Case Study 2: Client Onboarding SOP

### Context

**Situation:** Team members were onboarding clients inconsistently. Needed standardized process.

**Request:** "Create an SOP for onboarding new clients after agreement signed"

**Source Skill:** Command Protocol (Phase 2: Kickoff)

**Audience:** Internal team

---

### Process

#### Intake

**Clarification Questions Asked:**
- What triggers this process? → Signed agreement received
- Who's responsible? → Operations coordinator
- What's the desired outcome? → Client has access, kickoff scheduled, welcome sent

**Template Selected:** SOP

**Information Gathered:**
- Current process (observed)
- Tools used (CRM, Google Drive, scheduling)
- Pain points in current process
- Desired quality standards

---

#### Development

**Structure Applied:**
1. SOP header (effective date, version, status)
2. Purpose
3. Scope
4. Roles & Responsibilities
5. Procedure (9 steps)
6. Quality Checks
7. Exceptions
8. Related Documents
9. Revision History

**Procedure Steps Created:**

| Step | Action | Who | Timing |
|------|--------|-----|--------|
| 1 | Create client folder | Ops Coordinator | Within 4 hours |
| 2 | Add to CRM | Ops Coordinator | Same session |
| 3 | Send welcome email | Ops Coordinator | Within 4 hours |
| 4 | Schedule kickoff | Ops Coordinator | Within 24 hours |
| 5 | Gather prerequisites | Ops Coordinator | Before kickoff |
| 6 | Share access links | Ops Coordinator | Before kickoff |
| 7 | Confirm kickoff | Ops Coordinator | 24 hours before |
| 8 | Prep kickoff materials | Project Lead | Before kickoff |
| 9 | Log completion | Ops Coordinator | After kickoff |

**Quality Checks Added:**
- [ ] Client folder has all standard subfolders
- [ ] CRM record complete (name, contact, scope, value)
- [ ] Welcome email sent within 4 hours
- [ ] Kickoff scheduled within 2 business days
- [ ] All prerequisites gathered before kickoff

---

#### Quality Check

**SOP-Specific Checks:**
- [x] Each step has Who, What, When, How ✓
- [x] Quality checks are actionable ✓
- [x] Exception handling included ✓
- [x] Version tracking in place ✓

---

#### Outcome

**Final Document:** `TNDS_SOP_ClientOnboarding_2026-01-20.md`

**Time Spent:** 55 minutes

**Result:** Adopted by team, onboarding now consistent

---

### Lessons Learned

1. **Timing matters** — Adding specific timeframes (within 4 hours, within 24 hours) created accountability
2. **Quality checks drive compliance** — Checkable items more effective than vague standards
3. **Exceptions prevent workarounds** — Documenting "what if" scenarios reduced ad-hoc decisions

## Case Study 3: ASSESS Stage Cheat Sheet

### Context

**Situation:** Sales team needed a quick reference for running ASSESS calls consistently.

**Request:** "Create a cheat sheet for the ASSESS stage"

**Source Skill:** Direction Protocol

**Audience:** Sales team

---

### Process

#### Intake

**Clarification Questions Asked:**
- What's the call duration? → 30 minutes
- What must be covered? → Operational reality, current state, stakes, fit indicators
- What decisions result? → Continue to MAP or decline

**Template Selected:** Diagnostic/Cheat Sheet

**Information Gathered:**
- Direction Protocol ASSESS stage details
- Key questions from USER-MANUAL
- Red flags and green flags
- Scripts from existing materials

---

#### Development

**Structure Applied:**
1. Title and duration
2. Before You Start (setup)
3. Opening Script
4. Section 1: Operational Reality (10 min)
5. Section 2: Current State (10 min)
6. Section 3: Stakes & Fit (5 min)
7. Red Flags
8. Green Flags
9. Next Steps
10. Closing Script

**Scripts Written:**

**Opening:**
> "Thanks for making time. I've got about 30 minutes blocked. My goal today is to understand your operation well enough to know if we can help—and to give you enough information to know if you want us to. Fair enough?"

**Key Questions:**
> "Walk me through a typical day. What's the first fire you put out?"
> "Where's your data living right now—systems, spreadsheets, texts, paper?"
> "When was the last time you took a full day off? What happened?"

**Red Flags Documented:**
- [ ] Can't articulate specific pain (just "want to improve")
- [ ] No budget authority present
- [ ] Wants to skip to pricing ("just tell me what it costs")
- [ ] Blames team for all problems
- [ ] "We've tried everything"

**If 2+ red flags:** End call politely, do not schedule MAP

---

#### Quality Check

**Cheat Sheet-Specific Checks:**
- [x] Scripts are exact words ✓
- [x] Timing per section included ✓
- [x] Red flags clearly defined ✓
- [x] Action for flag count specified ✓

---

#### Outcome

**Final Document:** `TNDS_Diagnostic_AssessCheatSheet_2026-01-22.docx`

**Time Spent:** 30 minutes

**Result:** Sales team reports more consistent calls, better qualification

---

### Lessons Learned

1. **Exact scripts reduce variance** — Team stopped improvising, calls more predictable
2. **Red flag checklists prevent bad fits** — Clear criteria made declining easier
3. **Timing keeps calls on track** — Section timers prevented rabbit holes

## Case Study 4: Peak HVAC Proposal

### Context

**Situation:** Completed MAP session with Peak HVAC, a 10-technician HVAC company. Need to present findings and recommendation.

**Request:** "Create a proposal for Peak HVAC"

**Source Skill:** Direction Protocol (CHART stage)

**Audience:** Peak HVAC owner and office manager

---

### Process

#### Intake

**Information Gathered from MAP:**
- **Current State:** Job status unknown until technician calls in. Invoicing takes 3 days on average. Owner works 70+ hours/week, can't take a day off.
- **Root Causes:** Data scattered across texts, paper tickets, and one spreadsheet. No central system. Tribal knowledge.
- **Proposed Solution:** Command Center Build with data-command, financial-command, and sms-command modules
- **Scope:** Install visibility system, integrate with ServiceTitan, automate invoicing triggers
- **Investment:** $4,000 (40/30/30 payment)
- **Timeline:** 3 weeks

---

#### Development

**Structure Applied:**
1. Cover page
2. Executive Summary
3. Current Situation (from MAP)
4. Proposed Solution
5. Scope & Deliverables
6. Timeline
7. Investment & Payment Terms
8. Next Steps

**Executive Summary Written:**

> Peak HVAC has grown to 10 technicians, but the operation hasn't kept up. Job status is a mystery until someone calls in. Invoicing takes three days when it should take three minutes. And taking a day off means coming back to chaos.
>
> We propose a Command Center Build that gives you real-time visibility into every job, automates invoice creation, and lets you manage by exception instead of micromanagement.
>
> Investment: $4,000 | Timeline: 3 weeks | Payment: 40% at kickoff, 30% at build complete, 30% at launch.

**Key Writing Decisions:**

| Section | Approach |
|---------|----------|
| Current Situation | Used their words from MAP session |
| Proposed Solution | Tied each component to their specific pain |
| Deliverables | Listed exactly what they get, not generic features |
| Timeline | Showed weekly milestones, not just "3 weeks" |

---

#### Quality Check

**Proposal-Specific Checks:**
- [x] Client name spelled correctly ✓
- [x] MAP findings accurately reflected ✓
- [x] Scope boundaries clear ✓
- [x] Timeline realistic ✓
- [x] Payment terms specific ✓

---

#### Outcome

**Final Document:** `TNDS_Proposal_PeakHVAC_2026-01-25.docx`

**Time Spent:** 55 minutes

**Result:** Proposal accepted, agreement signed within one week

---

### Lessons Learned

1. **Use their words** — Quoting their pain points from MAP session made it personal
2. **Tie everything to pain** — Each deliverable connected to a specific problem they mentioned
3. **Show the timeline visually** — Week-by-week breakdown more compelling than duration only

## Case Study 5: financial-command Module Documentation

### Context

**Situation:** The financial-command module was built but not documented. Needed both client-facing and technical documentation.

**Request:** "Document the financial-command module"

**Source Skill:** Command Protocol (Phase 4-5)

**Audience:** Dual audience—clients (benefits) and technical team (implementation)

---

### Process

#### Intake

**Clarification Questions Asked:**
- Who needs this? → Both clients and technical team
- What's the module do? → Job costing, invoice automation, P&L visibility
- What systems does it integrate? → ServiceTitan, QuickBooks, Google Sheets

**Decision:** Create two documents—client-facing and technical

---

#### Development

**Client-Facing Document Structure:**
1. Module name and tagline
2. What This Module Does
3. Who It's For
4. Problems It Solves
5. Key Features
6. How It Works (simple)
7. Integration with Other Modules

**Technical Document Structure:**
1. Technical Overview
2. System Architecture
3. Data Model
4. API Integrations
5. Configuration
6. Testing
7. Troubleshooting

**Client-Facing Key Content:**

**What This Module Does:**
> financial-command gives you real-time visibility into the money side of your operation. See job costs as they happen, not 30 days later. Know which jobs made money and which ones bled you dry. Get invoices out the door same-day.

**Problems It Solves:**
- "I don't know if that job was profitable until the month-end report"
- "Invoicing takes days because we're chasing paperwork"
- "I can't tell which technicians cost us money"

**Technical Key Content:**

**Data Model:**
```
Job → JobCost → Invoice → Payment
 │
 └── LineItem → PartsCost
              → LaborCost
```

**API Integrations:**
- ServiceTitan API (job data, technician time)
- QuickBooks API (invoice creation, payment tracking)
- Google Sheets (reporting layer)

---

#### Quality Check

**Module-Specific Checks:**
- [x] Client doc uses benefits language ✓
- [x] Technical doc is implementation-complete ✓
- [x] Both reference same module accurately ✓

---

#### Outcome

**Final Documents:**
- `TNDS_Module_FinancialCommand_Client_2026-01-28.docx`
- `TNDS_Module_FinancialCommand_Technical_2026-01-28.md`

**Time Spent:** 45 minutes (both documents)

**Result:** Client version used in sales, technical version used for implementation

---

### Lessons Learned

1. **Two audiences, two documents** — Trying to serve both in one doc satisfies neither
2. **Same module, different language** — "Real-time job costing" (client) vs. "JobCost entity with LineItem children" (technical)
3. **Technical docs need specifics** — Data models, API details, configuration steps

## Patterns Across Case Studies

### Common Success Factors

| Factor | Impact |
|--------|--------|
| Clarify before writing | Avoided rework |
| Use source material | Accurate content |
| Apply correct template | Consistent structure |
| Voice check during writing | Caught issues early |
| Quality checklist before delivery | Zero revision requests |

### Common Time Investment

| Document Type | Typical Time |
|---------------|--------------|
| Service Document | 30-40 minutes |
| SOP | 50-60 minutes |
| Cheat Sheet | 25-35 minutes |
| Proposal | 50-60 minutes |
| Module Documentation | 40-50 minutes |

### Voice Transformation Patterns

| Before (Common Error) | After (Correct) |
|----------------------|-----------------|
| "Enables visibility" | "You'll see" |
| "Optimizes workflow" | "Cuts time" |
| "Leverages integration" | "Connects to" |
| "Stakeholder alignment" | "Everyone on the same page" |
