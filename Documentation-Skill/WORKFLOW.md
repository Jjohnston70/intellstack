# Documentation Skill Workflow

**Version:** 1.0
**Last Updated:** January 29, 2026
**Purpose:** Step-by-step process for creating TNDS-branded documentation from request to delivery

## Overview

The Documentation workflow takes any documentation request through a structured process that ensures consistency, quality, and brand alignment. This workflow applies whether you're creating a client proposal, internal SOP, or diagnostic cheat sheet.

## Phase 1: Intake (5-10 minutes)

### Step 1.1: Clarify the Request

**Purpose:** Understand exactly what's needed before starting.

**Questions to Ask:**

| Category | Question |
|----------|----------|
| **Audience** | Who will read this document? (Client, team, technical) |
| **Purpose** | What action should the reader take after reading? |
| **Format** | Do you need Word (.docx) or Markdown (.md)? |
| **Urgency** | When is this needed? |
| **Context** | Is this part of a larger project? Which skill triggered this? |

**Clarification Checklist:**
- [ ] Audience identified
- [ ] Purpose clear
- [ ] Format decided
- [ ] Timeline understood
- [ ] Source skill identified (if applicable)

---

### Step 1.2: Select Template

**Template Decision Tree:**

```
Is this for a client?
├── YES → Is it a service description?
│         ├── YES → CLIENT-FACING SERVICE DOCUMENT
│         └── NO → Is it a proposal?
│                  ├── YES → PROPOSAL TEMPLATE
│                  └── NO → Is it an agreement?
│                           ├── YES → AGREEMENT TEMPLATE
│                           └── NO → DIAGNOSTIC CHEAT SHEET
│
└── NO → Is it a repeatable process?
          ├── YES → Does it need compliance tracking?
          │         ├── YES → SOP TEMPLATE
          │         └── NO → INTERNAL PROCESS DOCUMENT
          │
          └── NO → Is it technical?
                    ├── YES → TECHNICAL SPECIFICATION
                    └── NO → INTERNAL PROCESS DOCUMENT
```

**Quick Template Reference:**

| Template | Use For | Format |
|----------|---------|--------|
| Client-Facing Service | Service descriptions | Word |
| Proposal | Project proposals | Word |
| Agreement | Contracts | Word |
| Diagnostic Cheat Sheet | Assessment tools | Word |
| SOP | Repeatable processes | Markdown |
| Internal Process | How-to guides | Word or Markdown |
| Technical Spec | System docs | Markdown |

---

### Step 1.3: Gather Information

**What to Collect (varies by template):**

**For Client-Facing Documents:**
- [ ] Service/product details
- [ ] Target audience characteristics
- [ ] Pain points addressed
- [ ] Deliverables list
- [ ] Pricing and timeline
- [ ] Prerequisites

**For Internal Documents:**
- [ ] Process steps
- [ ] Roles involved
- [ ] Tools required
- [ ] Decision points
- [ ] Common issues

**For Proposals:**
- [ ] Client name and context
- [ ] MAP session findings
- [ ] Proposed solution details
- [ ] Scope boundaries
- [ ] Pricing breakdown

**Information Sources:**
- Direction Protocol outputs (for client docs)
- World Model Mapper outputs (for process docs)
- Bearing Check outputs (for decision docs)
- Subject matter expert input
- Existing templates and examples

## Phase 2: Development (15-60 minutes)

### Step 2.1: Structure Content

**Apply the Chosen Template:**

1. Copy the template structure from SKILL.md
2. Create all required sections
3. Note any sections that need modification

**Structuring Rules:**
- Front-load important information
- Use headers to create clear hierarchy
- Plan bullet points vs. paragraphs
- Identify where tables improve clarity

---

### Step 2.2: Write Content

**Writing Process:**

1. **Draft section by section** — Don't write linearly; tackle sections in order of importance
2. **Apply TNDS voice** — Direct, no-fluff, outcomes-focused
3. **Use key phrases** — Reference messaging from SKILL.md
4. **Stay specific** — No vague generalities

**Voice Quick Check:**

| Ask Yourself | Good Answer |
|--------------|-------------|
| Am I leading with outcomes? | "You'll be able to..." |
| Am I being specific? | Numbers, examples, concrete details |
| Am I being direct? | Short sentences, active voice |
| Am I avoiding jargon? | Plain language, no buzzwords |

**Writing Timing by Document Type:**

| Document Type | Typical Writing Time |
|---------------|---------------------|
| Service document | 20-30 minutes |
| SOP | 30-45 minutes |
| Cheat sheet | 15-25 minutes |
| Proposal | 45-60 minutes |
| Agreement | 30-45 minutes |
| Module documentation | 20-30 minutes |

---

### Step 2.3: Format Document

**For Word Documents:**

1. **Header Setup**
   - Insert TNDS logo (centered)
   - Logo height: ~0.8-1.0 inches

2. **Apply Typography**
   - H1: Arial, 36pt, Bold, Navy (#1a3a5c)
   - H2: Arial, 28pt, Bold, Teal (#3d8eb9)
   - H3: Arial, 24pt, Bold, Navy
   - Body: Arial, 11pt, Regular, Black

3. **Apply Spacing**
   - After H1: 12pt
   - After H2: 10pt
   - After paragraphs: 10pt
   - After bullets: 6pt
   - Line spacing: 1.15

4. **Footer Setup**
   - Line 1: Jacob Johnston | 719-204-6365 | jacob@truenorthstrategyops.com
   - Line 2: True North Data Strategies • Colorado Springs, CO (italicized)
   - Font: Arial, 9pt, Navy, Centered

**For Markdown Documents:**

1. **Use proper header hierarchy**
   ```markdown
   # H1 Title
   ## H2 Section
   ### H3 Subsection
   ```

2. **Format lists consistently**
   ```markdown
   - Bullet item
     - Sub-item (2-space indent)

   1. Numbered item
      1. Sub-item (3-space indent)
   ```

3. **Use blockquotes for scripts**
   ```markdown
   > "Exact words to say"
   ```

4. **Add frontmatter if needed**
   ```yaml
   ---
   title: Document Title
   author: True North Data Strategies
   date: 2026-01-29
   type: sop
   ---
   ```

## Phase 3: Quality Assurance (5-10 minutes)

### Step 3.1: Self-Review Checklist

**Content Quality:**
- [ ] Voice matches TNDS tone (direct, no-fluff)
- [ ] Uses outcomes language ("you'll be able to...")
- [ ] Specific and concrete (no vague generalities)
- [ ] Client-appropriate language level
- [ ] Key phrases used correctly
- [ ] No jargon without definition
- [ ] No emojis

**Structure Quality:**
- [ ] Follows appropriate template
- [ ] Logical flow and organization
- [ ] Headers create clear hierarchy
- [ ] Sections appropriate length
- [ ] Bullets for lists, prose for single points
- [ ] Tables where they improve clarity

---

### Step 3.2: Voice Check

**Read the document aloud.** Fix anything that sounds:
- Corporate or buzzword-y
- Passive or indirect
- Vague or generic
- Apologetic or hedging

**Voice Transformation Examples:**

| Before (Wrong) | After (Right) |
|----------------|---------------|
| "We strive to provide..." | "You get..." |
| "Our solution leverages..." | "This shows you..." |
| "Stakeholder alignment" | "Everyone on the same page" |
| "We hope to..." | "We will..." |

---

### Step 3.3: Brand Check

**For Word Documents:**
- [ ] Logo present and centered in header
- [ ] Footer formatted correctly (2 lines)
- [ ] H1 is Navy (#1a3a5c)
- [ ] H2 is Teal (#3d8eb9)
- [ ] Font is Arial throughout
- [ ] Spacing is consistent
- [ ] No random colors or fonts

**For Markdown Documents:**
- [ ] Proper header hierarchy
- [ ] Consistent formatting
- [ ] Links work
- [ ] Code blocks formatted
- [ ] Tables display correctly

---

## Phase 4: Finalization (5 minutes)

### Step 4.1: Final Formatting

**Word Document Final Pass:**
- [ ] Check page breaks (no orphaned headers)
- [ ] Verify no widows/orphans (single lines at top/bottom)
- [ ] Confirm margins are 1" all sides
- [ ] Test print preview

**Markdown Document Final Pass:**
- [ ] Preview rendered output
- [ ] Check all links
- [ ] Verify code blocks render
- [ ] Confirm tables display

---

**Naming Convention:**

```
TNDS_[Type]_[Name]_[Date].ext
```

**Type Codes:**

|------|---------------|

## Phase 5: Delivery

### Step 5.1: Share Document

**Client Documents:**
1. Attach to email (see EMAIL-TEMPLATES.md)
2. Use clear subject line
3. Summarize key points in email body
4. Set expectation for next step

**Internal Documents:**
1. Save to appropriate folder
2. Notify relevant team members
3. Add to training queue if applicable
4. Schedule review if SOP

---

### Step 5.2: Confirm Receipt

**For Client Documents:**
- Follow up within 2 business days if no response
- Ask if they have questions
- Confirm receipt and understanding

**For Internal Documents:**
- Verify team members can access
- Confirm they've reviewed
- Answer questions

---

### Step 5.3: Handle Feedback

**Revision Request Received:**

1. **Acknowledge** — Respond within 24 hours
2. **Clarify** — Understand exactly what needs to change
3. **Revise** — Make changes following same workflow
4. **Re-deliver** — Send updated version with change summary
5. **Version** — Save previous version if significant changes

**Naming for Revisions:**
- Minor updates: Overwrite original
- Major revisions: `TNDS_Type_Name_Date_v2.ext`

## Document Type Workflows

### Service Document Workflow

```
Trigger: New service needs documentation
    │
    ├── 1. Identify service details (pricing, timeline, deliverables)
    ├── 2. Define ICP for this service
    ├── 3. List pain points it addresses
    ├── 4. Use CLIENT-FACING SERVICE template
    ├── 5. Write outcomes-focused content
    ├── 6. Format as Word document
    ├── 7. Quality check (voice, brand)
    └── 8. Save to template library
```

**Time:** 30-45 minutes

---

### SOP Workflow

```
Trigger: Process needs standardization
    │
    ├── 1. Document current process (observe or interview)
    ├── 2. Identify roles and responsibilities
    ├── 3. Define quality checkpoints
    ├── 4. Use SOP template
    ├── 5. Write step-by-step procedure
    ├── 6. Add decision points and exceptions
    ├── 7. Format as Markdown
    ├── 8. Review with process owner
    └── 9. Save to SOPs folder
```

**Time:** 45-60 minutes

---

### Proposal Workflow

```
Trigger: MAP session complete, client ready for proposal
    │
    ├── 1. Gather MAP session outputs
    ├── 2. Define proposed solution
    ├── 3. Create scope and timeline
    ├── 4. Calculate pricing
    ├── 5. Use PROPOSAL template
    ├── 6. Write client-specific content
    ├── 7. Format as Word document
    ├── 8. Quality check (accuracy, brand)
    ├── 9. Save to client folder
    └── 10. Deliver via email
```

**Time:** 45-60 minutes

---

### Cheat Sheet Workflow

```
Trigger: Stage or process needs quick reference tool
    │
    ├── 1. Identify the process to document
    ├── 2. Gather scripts, questions, timing
    ├── 3. Define red flags and green flags
    ├── 4. Use DIAGNOSTIC template
    ├── 5. Write exact scripts and criteria
    ├── 6. Format as Word document
    ├── 7. Test with actual use case
    └── 8. Save to Diagnostic folder
```

**Time:** 25-35 minutes

---

### Module Documentation Workflow

```
Trigger: Command module needs documentation
    │
    ├── 1. Determine audience (client vs. technical)
    ├── 2. Gather module specifications
    ├── 3. If client-facing: focus on benefits
    ├── 4. If technical: focus on implementation
    ├── 5. Use MODULE-DOCUMENTATION template
    ├── 6. Write appropriate content
    ├── 7. Format (Word for client, MD for technical)
    └── 8. Save to module folder
```

**Time:** 20-30 minutes
