# ARO Assessment Quality Review Checklist

**Version:** 1.0
**Last Updated:** January 28, 2026
**Purpose:** Ensure consistent, high-quality deliverables before client delivery

---

## Overview

Quality review happens in 4 stages:

| Stage | Who | When | Duration |
|-------|-----|------|----------|
| 1. Self-Review | Assessment creator | Day 6 | 30-60 min |
| 2. Peer Review | Jacob or senior team member | Day 6 | 30-45 min |
| 3. Revision | Assessment creator | Day 6 (if needed) | 15-30 min |
| 4. Final Approval | Jacob | Day 7 | 10-15 min |

**Rule:** No deliverables go to client without completing all 4 stages.

---

## STAGE 1: Self-Review (Day 6)

**Who:** Assessment creator (sales guy or Jacob)
**Time Required:** 30-60 minutes
**Purpose:** Catch obvious errors and ensure completeness before peer review

### Instructions

Go through each checklist below. Mark items complete only when you've verified them. If something is wrong, fix it before moving to peer review.

---

### Context Architecture Report Checklist

**Structure & Format:**
- [ ] Document is 10-15 pages (not too short, not bloated)
- [ ] Executive Summary is exactly 1 page
- [ ] All section headers present and properly formatted
- [ ] Page numbers included
- [ ] TNDS branding consistent (if using branded template)

**Executive Summary (Page 1):**
- [ ] Overall score displayed prominently (X.X / 5.0)
- [ ] Top 3 friction points listed with brief descriptions
- [ ] Recommended investment range included
- [ ] Key finding or insight highlighted

**Current State Assessment (3-4 pages):**
- [ ] All 6 operational areas scored
- [ ] Each score has specific justification (not generic)
- [ ] Examples from discovery call cited for each score
- [ ] Current state accurately reflects what you learned
- [ ] No contradictions between sections

**Recommendations (4-5 pages):**
- [ ] Each recommendation tied to a specific finding
- [ ] Recommendations are actionable (not vague platitudes)
- [ ] Priority order makes logical sense
- [ ] No conflicting recommendations
- [ ] Effort levels noted (Quick Win / Medium / Major)

**Visual Diagrams (2-3 pages):**
- [ ] Current state diagram included
- [ ] Future state diagram included
- [ ] Priority matrix or similar visualization included
- [ ] Diagrams are readable and not cluttered
- [ ] Diagrams match the text content

**General Quality:**
- [ ] Client name spelled correctly throughout
- [ ] No spelling or grammar errors
- [ ] TNDS voice: direct, no-fluff, outcome-focused
- [ ] No placeholder text left in ([brackets] or Lorem ipsum)
- [ ] Dates are accurate

---

### Implementation Roadmap Checklist

**Structure & Format:**
- [ ] Document is 3-5 pages
- [ ] Clear section breaks between Quick Wins, Phase 1, Phase 2
- [ ] Timeline visualization included (Gantt or similar)

**Quick Wins Section (0-30 days):**
- [ ] Each Quick Win is actually quick (<10 hours effort)
- [ ] 3-5 Quick Wins identified (not too many, not too few)
- [ ] Each has specific, measurable outcome
- [ ] No dependencies blocking Quick Wins
- [ ] Effort estimates included for each

**Phase 1 Section (30-90 days):**
- [ ] Items properly sequenced (dependencies respected)
- [ ] Each item has clear description of what's done
- [ ] Resources needed are noted
- [ ] Timeline estimates are realistic
- [ ] Dependencies on Quick Wins completion noted

**Phase 2 Section (90-180 days):**
- [ ] Larger structural changes identified
- [ ] Budget estimates included
- [ ] Dependencies on Phase 1 noted
- [ ] Long-term outcomes described

**Implementation Paths:**
- [ ] DIY path addressed with honest assessment
- [ ] TNDS-led path addressed with pricing
- [ ] Hybrid option mentioned if applicable
- [ ] $500 assessment credit mentioned for TNDS path

**General Quality:**
- [ ] Timeline estimates are realistic for this client
- [ ] Budget estimates are in correct range
- [ ] No placeholder text remaining
- [ ] Consistent with Context Architecture Report findings

### Context Graph Checklist

**Structure & Completeness:**
- [ ] All 9 sections present
- [ ] Version marked as 1.0 - Assessment Baseline
- [ ] Last Updated date is accurate

**Section 1: Entity Profile**
- [ ] Business name correct
- [ ] Contact information accurate
- [ ] Industry/business type noted
- [ ] Relationship context captured

**Section 2: Current Tech Stack**
- [ ] All systems mentioned in discovery listed
- [ ] Access levels noted where known
- [ ] Integration status noted
- [ ] Not just a copy of intake questionnaire

**Section 3: Business Context**
- [ ] Pain points listed and ranked by severity
- [ ] Success metrics defined (what client wants to achieve)
- [ ] Constraints noted (budget, timeline, team capacity)
- [ ] Key stakeholders identified

**Section 4: Project History**
- [ ] Assessment completion noted
- [ ] Discovery call date recorded
- [ ] Assessment tier noted

**Section 5: Communication Log**
- [ ] Discovery call summarized
- [ ] Key decisions or agreements noted
- [ ] Handoff call scheduled date noted

**Section 6: Operational Rules**
- [ ] Client preferences captured from discovery
- [ ] "Do always" rules noted
- [ ] "Never do" constraints noted

**Section 7: Financial Context**
- [ ] Assessment contract details correct
- [ ] Tier and amount accurate
- [ ] Potential implementation range noted

**Section 8: Artifact Index**
- [ ] Folder location noted
- [ ] Document inventory listed
- [ ] All 4 deliverables referenced

**Section 9: Quick Reference**
- [ ] One-liner summary is accurate and useful
- [ ] Current priority clear
- [ ] Red flags noted (if any)

**General Quality:**
- [ ] NOT a generic template — clearly customized
- [ ] Information matches discovery findings
- [ ] Would be useful to AI agent working with this client
- [ ] No placeholder text remaining

---

### Documentation System Recommendation Checklist

**Content:**
- [ ] Tier recommendation stated clearly (1, 2, or 3)
- [ ] Justification provided based on client needs
- [ ] Tier differences explained
- [ ] Recommendation matches client complexity

**Accuracy:**
- [ ] Tier 1 recommended only for simple, single-workflow clients
- [ ] Tier 2 recommended for most implementation clients
- [ ] Tier 3 reserved for regulated/compliance-heavy clients
- [ ] Recommendation consistent with Context Architecture Report

### Cross-Document Consistency Check

- [ ] Top 3 friction points consistent across documents
- [ ] Client name spelled the same everywhere
- [ ] Dates are consistent
- [ ] Pricing is consistent
- [ ] No contradictions between documents

## STAGE 2: Peer Review (Day 6)

**Who:** Jacob (if sales guy did assessment) OR senior team member (if Jacob did assessment)
**Time Required:** 30-45 minutes
**Purpose:** Fresh eyes catch what creator missed; ensure quality bar is met

### Instructions for Reviewer

You're checking for quality, accuracy, and client-readiness. You haven't been in the weeds — use your fresh perspective to catch issues.

---

### Scoring Review

**Score Validity:**
- [ ] Each score feels accurate based on justification provided
- [ ] Scores aren't inflated (saying 4 when clearly a 2)
- [ ] Scores aren't deflated (underselling to upsell)
- [ ] Justifications cite specific examples, not generic statements

**Red Flags to Check:**
- [ ] No score justified with "some room for improvement" (too vague)
- [ ] No score justified with "could be better" (too vague)
- [ ] No scores that contradict the evidence presented

- [ ] Priority order makes sense (high impact first)

**Red Flags to Check:**

- [ ] No recommendations that contradict each other

## Common Quality Issues to Avoid

### Scoring Issues
- **Generic justifications:** "Could be improved" → Always cite specific examples
- **Inflated scores:** Giving 4s when evidence shows 2s → Be honest
- **Missing evidence:** Score without justification → Always explain why

### Recommendation Issues
- **Boilerplate:** Same recommendations for every client → Customize
- **Too many:** 20 recommendations overwhelms → Focus on top 5-10
- **Too vague:** "Improve documentation" → Specify what and how

### Context Graph Issues
- **Template copy:** Generic sections not filled in → Customize everything
- **Missing info:** Blank sections → Fill from discovery notes
- **Outdated:** Info doesn't match current state → Verify accuracy

### Roadmap Issues
- **Unrealistic timelines:** 30 days for major project → Be honest
- **Wrong pricing:** Estimates way off market → Use pricing guide
- **Missing paths:** No DIY option → Always include both paths
