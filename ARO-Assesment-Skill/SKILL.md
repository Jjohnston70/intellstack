# ARO Assessment™ Skill

A structured framework for evaluating how "AI-legible" a client's operations are and producing deliverables that help them become agent-ready.

## Core Concept

Most businesses organize information for humans who can infer, guess, and ask follow-up questions. AI agents need:
- Explicit context (not implied)
- Consistent structure (not ad-hoc)
- Clear decision logic (not "use your judgment")
- Accessible data (not locked in someone's head)

ARO Assessment identifies gaps between current operations and agent-ready operations.

## Assessment Workflow

### Phase 1: Discovery (Day 1-2)

**Pre-Call:**
- Client completes ARO Intake Questionnaire
- Review any documentation shared
- Form initial hypothesis on friction points

**Discovery Call (90 min):**
Walk through these question categories:

*Current AI Usage:*
- What AI tools in use? Where do they work/fail?
- What do they wish AI could do?

*Information Architecture:*
- Where does data live? (CRM, spreadsheets, email, paper)
- File organization quality?
- What exists only in someone's head?

*Process Documentation:*
- Written SOPs exist?
- How do new people learn "how things work"?
- What decisions require specific person's judgment?

*Decision Logic:*
- Walk through most common workflow
- What criteria determine different outcomes?
- How do they prioritize?

*Technology Landscape:*
- What systems don't talk to each other?
- Where is data entered twice?
- What breaks if key person unavailable?

### Phase 2: Analysis (Day 3-4)

**Context Friction Audit**

Score each area 1-5:

| Area | 1 (Chaotic) | 3 (Functional) | 5 (Agent-Ready) |
|------|-------------|----------------|-----------------|
| File Organization | No structure; random names | Basic folders; inconsistent naming | Hierarchical; consistent; metadata tagged |
| Process Documentation | Tribal knowledge only | Some SOPs; outdated | Current, explicit, step-by-step |
| Data Structure | Scattered; duplicates | Centralized but messy | Single source of truth; clean relationships |
| Decision Logic | "Ask Bob" | General guidelines | Explicit if/then rules documented |
| Communication Records | Email chaos | Some logging | Structured history; searchable |
| Access & Permissions | Unknown who has what | Basic controls | Clear matrix; API access where needed |

**Gap Analysis**

For each area scoring below 4:
- What friction does this cause for AI agents?
- What would "fixed" look like?
- Difficulty: Quick win / Medium effort / Major project
- Business impact of fixing?

**Tool Compatibility**
- Unused capabilities in current tools?
- Available integrations not configured?
- APIs available for automation?

### Phase 3: Deliverables (Day 5-7)

Generate four deliverables:

**1. Context Architecture Report (10-15 pages)**
- Executive Summary (1 page): Overall score, top 3 friction points, recommended investment
- Current State Assessment (3-4 pages): Detailed scoring, specific examples, how gaps manifest
- Recommendations (4-5 pages): Proposed hierarchy, documentation standards, data improvements
- Visual Diagrams (2-3 pages): Current vs. future state maps, priority matrix

**2. Implementation Roadmap (3-5 pages)**
- Quick Wins (0-30 days, <10 hours each)
- Phase 1 (30-90 days): Medium complexity, dependencies, resources
- Phase 2 (90-180 days): Larger structural changes, budget estimates

**3. Context Graph Template (Customized)**
- Pre-populated for their business
- See references/context-graph-template.md

**4. Documentation System Recommendation**
- Tier 1: Memory-Bank Only (simple automations)
- Tier 2: Memory-Bank + SpecKit (development projects)
- Tier 3: Memory-Bank + SpecKit Production (regulated industries)

### Phase 4: Handoff (Day 7)

**Handoff Call (60 min):**
- Walk through report
- Explain scoring and recommendations
- Review roadmap together
- Discuss implementation options (DIY vs. TNDS-led)

**Post-Handoff:**
- 30-day email support
- One follow-up call if needed
- Priority scheduling for implementation

## Scoring Guidelines

### File Organization
- **1:** Files named "doc1.docx", no folders, desktop chaos
- **2:** Some folders but inconsistent, can't find things without searching
- **3:** Folder structure exists, naming mostly consistent, findable with effort
- **4:** Clear hierarchy, naming conventions followed, metadata sometimes used
- **5:** Fully organized, consistent naming, tagged/searchable, self-documenting

### Process Documentation
- **1:** "Ask Sarah" is the documentation
- **2:** A few outdated docs exist somewhere
- **3:** Key processes documented but not maintained
- **4:** Most processes documented, reviewed quarterly
- **5:** All processes documented, version-controlled, actively maintained

### Data Structure
- **1:** Same customer in 5 spreadsheets with different info
- **2:** Primary system exists but people work around it
- **3:** Central system used but data quality issues
- **4:** Clean central system, occasional sync issues
- **5:** Single source of truth, clean data, clear relationships

### Decision Logic
- **1:** Every decision requires finding the right person
- **2:** Some rules exist but mostly tribal knowledge
- **3:** Guidelines exist but judgment calls common
- **4:** Most decisions have documented criteria
- **5:** Decision trees documented, exceptions handled explicitly

### Communication Records
- **1:** Important conversations lost in email/text threads
- **2:** Some notes taken but not searchable
- **3:** CRM/system captures some history
- **4:** Most interactions logged, searchable
- **5:** Complete history, structured, AI-queryable

### Access & Permissions
- **1:** Shared passwords, unclear who has access to what
- **2:** Individual accounts but admin access unclear
- **3:** Basic permission structure, no API strategy
- **4:** Clear permissions, some API access configured
- **5:** Full access matrix, API keys managed, automation-ready

## Context Graph Structure

When creating Context Graphs, include these sections:

1. **Entity Profile** - Contact info, business identity, relationship context
2. **Current Tech Stack** - Systems, access levels, integrations
3. **Business Context** - Pain points, success metrics, constraints, stakeholders
4. **Project History** - Completed work, active streams, decisions made
5. **Communication Log** - Recent interactions, open threads, sensitive topics
6. **Operational Rules** - Do always, never do, client preferences
7. **Financial Context** - Contract details, upsell opportunities
8. **Artifact Index** - Document locations, credentials
9. **Quick Reference** - One-liner summary, current priority, red flags

See templates/context-graph-template.md for full structure.
